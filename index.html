프로젝트 개요

프로젝트 개요

카드 결제에서 같은 요청이 네트워크 재시도로 두 번 들어와도 결제가 '정확히 한 번만' 처리되도록, 그리고 여러 요청이 동시에 같은 잔액을 갱신해도 정합성이 깨지지 않도록 설계한 백엔드 시스템입니다. 이중결제 방지와 동시성 상황의 잔액 정합성이라는 금융 시스템의 핵심 문제 두 가지를, 멱등성 키·낙관적 락·append-only 원장 설계로 풀어내고 결정적 동시성 테스트로 검증했습니다. 결제 승인, 취소·부분 환불, 조회 API까지 핵심 결제 흐름이 구현되어 있습니다.

핵심 기능 & 로직

  • ✦

    멱등성 키 2단 방어 — idempotencyKey 코드 검사(1차, UX·효율)와 DB unique 제약(최후 방어선)을 분리 설계해, 동시 요청이 코드 검사를 통과해도 두 번째 INSERT가 제약 위반으로 롤백되게 함

  • ✦

    낙관적 락 동시성 제어 — Account.version(@Version)으로 두 결제가 같은 잔액을 덮어쓰는 lost update를 감지, 충돌 트랜잭션 전체를 롤백해 '일부만 반영' 상태를 원천 차단

  • ✦

    부분 취소 오버리펀드 방지 — Payment에도 @Version을 두고 refundedAmount를 직접 갱신해, 동시 취소가 원 결제 금액을 초과하는 레이스를 저장 시점에 차단

  • ✦

    단일 트랜잭션 원자성 — '잔액 확인→차감→원장 기록→상태 변경'을 하나의 @Transactional로 묶어 부분 성공 자체가 존재하지 않게 (실패는 완전히 실패)

  • ✦

    Append-only 원장 설계 — LedgerEntry는 수정 없이 반대 방향 CREDIT 행을 새로 쌓아, 계좌의 자금 흐름을 시간순 그대로 복원 가능하게 (회계/카드사 원장 원칙)

  • ✦

    전역 예외 처리 — @RestControllerAdvice 한 곳에 에러 정책을 모으고 상태코드를 책임 소재로 분리(404/409/400/500), 스택트레이스는 로그에만 남기고 클라이언트엔 통일된 형식만 응답

  • ✦

    명령·조회 책임 분리 — 쓰기(pay/cancel)와 읽기(QueryService)를 클래스로 분리하고 조회엔 @Transactional(readOnly=true)를 적용해 안전성과 성능 확보

  • ✦

    실제 MySQL 기반 동시성 검증 — Testcontainers로 MySQL 8.4 컨테이너를 띄워 동일 멱등성 키의 동시 요청을 10·50·100 스레드로 총 960건 재현, 중복 결제 0건 · 잔액 오차 0원 · 원장 정확히 1건을 실측 검증

이슈

두 결제 요청이 거의 동시에 들어오면 둘 다 '기존 결제 없음'으로 판단해 이중결제 통과

해결 → idempotencyKey에 DB unique 제약을 걸어 코드 검사가 뚫려도 2차 INSERT가 롤백되게, '1차 방어=코드, 최후 방어=DB 제약' 2계층 구조로 설계

이슈

동시 부분 취소 시 두 요청이 같은 refundedAmount를 읽어 원 결제액 초과 취소(오버리펀드) 발생

해결 → Payment에 @Version을 추가해 두 번째 취소를 저장 시점에 차단, 스레드 타이밍에 의존하지 않는 결정적 테스트로 재현·검증

이슈

테스트를 H2 인메모리 DB로만 돌리고 있어, 멱등성의 최후 방어선인 DB unique 제약이 실제 MySQL에서도 동일하게 동작하는지 검증되지 않음

해결 → Testcontainers로 실제 MySQL 8.4 컨테이너를 띄워 테스트 전환. 이 과정에서 시드 데이터의 H2 전용 함수(DATEADD) 의존이 드러나, 테스트가 인메모리 DB에 종속돼 있었음을 확인하고 제거

이슈

동시 요청을 10 → 100으로 늘려도 충돌 실패가 항상 정확히 9건으로 고정

해결 → HikariCP 커넥션 풀 크기(기본 10)가 동시 트랜잭션 수를 제한해 실제 경합 창이 풀 크기로 결정됨을 규명. 나머지 요청은 대기 중 승자가 커밋을 마쳐 1차 방어(선조회)에 걸려 재생 응답을 받는 구조임을 실측으로 확인

프로젝트 개요

거동이 불편한 어르신과 교통약자를 위한 배리어프리 길찾기 서비스입니다. ORS(OpenRouteService) API로 최단 경로와 계단 회피 경로 두 가지를 받아온 뒤, 좌표별 고도값으로 구간마다 경사도(%)를 직접 계산하고 계단 여부와 결합해 보행 비용을 산출합니다. 이 비용을 비교해 더 안전한 경로를 추천하며, 저는 프론트엔드(React + Vite)와 백엔드(Node.js + Express) 개발을 모두 담당했습니다.

핵심 기능 & 로직

  • ✦

    경사도 계산 엔진 — ORS가 반환한 좌표별 고도(elevation) 데이터로 구간별 경사도(%)를 고도차 ÷ 구간거리로 서버에서 직접 산출

  • ✦

    보행 비용 채점 & 경로 비교 — ORS에 최단 경로·계단 회피 경로를 각각 요청한 뒤, 경사도·계단 여부에 가중치를 곱해 보행 비용을 매기고 더 안전한 경로를 추천

  • ✦

    음성 길안내 — Web Speech API 기반 TTS로 경로 안내를 음성으로 제공해 접근성 강화

이슈

해커톤 종료 후 코드를 재검증하던 중, 12km 경로에서 최대 경사 283%가 검출됨. 원인은 분모가 아니라 분자였음 — 거리 0 구간은 방어했지만, ORS 고도 원천인 SRTM(90m 격자)이 한강변에서 6.7m 구간에 19m의 고도 점프를 반환했고, 이 노이즈 구간이 ×4.0 위험 가중치를 받아 경로 비용이 오염되며 추천 결과 자체가 뒤집힘

해결 → 세계 최급경사 도로(34.8%)를 근거로 40% 초과를 데이터 오류로 처리하고, 격자 크기의 절반인 ±45m 삼각 가중 평균으로 고도를 스무딩. 최대 경사 283.1% → 30.5%, 정상 경로는 7.4%로 변화 없음을 실측 검증(재현 테스트 9종 추가)

이슈

도착 판정을 '현재 위치에서 가장 가까운 경로 구간'으로 계산했는데, 경사 회피 우회 경로는 왔던 길 근처로 되돌아오는 구간이 많아 출발 지점이 복귀 구간에 오매칭됨 — 핵심 기능(우회)이 핵심 판정(도착)을 깨뜨리는 구조로, 남은 거리 451m가 14m로 계산되어 걷기 시작하자마자 "도착했습니다" 음성이 나감

해결 → 직전 매칭 구간 기준 전방 60m 윈도우로 진행 순서를 강제, 단 이탈 시 영영 복구 불가한 부작용을 막기 위해 35m 초과 이탈 시 전체 재탐색으로 리셋. 451.5m 정상 반환 확인. 첫 구현(구간 개수 기준 윈도우)이 재현 테스트에 걸려 거리 기준으로 재설계한 과정에서, 테스트를 먼저 만든 것이 실제로 작동함

프로젝트 개요

지금 보고 계신 이 포트폴리오 웹사이트 자체가 하나의 프로젝트입니다. React나 Vue 같은 프레임워크 없이 순수 HTML/CSS/JS만으로 웜톤 컬러 시스템, 아코디언, 확장형 카드 등의 인터랙션을 직접 설계하고 구현했습니다. 컴포넌트 없이도 유지보수 가능한 구조를 만들기 위해 BEM 네이밍 규칙과 CSS 변수 기반 디자인 시스템을 일관되게 적용했습니다.

핵심 기능 & 로직

  • ✦

    CSS 변수 기반 디자인 시스템 — --coral, --honey 등 변수 하나로 전체 톤앤매너를 일괄 관리해 확장·수정 용이성 확보

  • ✦

    확장형(Expanding) 프로젝트 카드 — 팝업 없이 카드 자체가 그 자리에서 펼쳐지는 max-height/opacity 트랜지션과, 한 번에 하나만 열리도록 제어하는 아코디언 그룹 로직 구현

  • ✦

    반응형 그리드 레이아웃 — Hero/About/Projects 각 섹션을 CSS Grid와 clamp()로 구성해 데스크톱부터 모바일까지 최소한의 미디어쿼리로 대응

  • ✦

    부드러운 스크롤 내비게이션 — 고정 헤더 높이를 보정한 커스텀 스크롤 로직으로 앵커 이동 시 콘텐츠 잘림 방지

이슈

한 카드 안에 카드 전체 확장과 Troubleshooting 아코디언, 두 종류의 열고 닫힘 상태가 공존하며 로직이 얽힘

해결 → 카드 확장 여부와 내부 아코디언 여부를 각각 독립된 aria-expanded 상태로 분리해 관리, 상호 간섭 없이 동작하도록 구현

이슈

실제 이미지 없이 이모지만으로 일러스트를 표현하다 보니 화면이 정적으로 느껴짐

해결 → absolute로 배치한 도형 요소에 keyframe 애니메이션(floaty)을 적용해 은은한 움직임을 부여, 따뜻하고 생동감 있는 느낌으로 개선

이슈

프로젝트 카드 5개에 유튜브 iframe을 정적으로 임베드했더니, 접힌 카드(max-height: 0)에서도 브라우저가 iframe을 전부 로드해 열어보지도 않은 영상 플레이어가 초기 로드에서 요청 44건·1.1MB를 소모. DevTools Network 실측으로 발견

해결 → 외부 라이브러리 없이 파사드 패턴을 바닐라 JS로 직접 구현 — 클릭 전엔 정적 썸네일과 재생 버튼만 두고, 클릭 시에만 autoplay iframe을 주입. 초기 요청 56→10건(−82%), 전송량 −1.1MB, 유튜브 요청 0건을 재측정으로 검증

이슈

Lighthouse 실측 결과 총 전송량 4.2MB 중 3MB(72%)가 Pretendard 폰트였음. 페이지에 실제 쓰이는 글자는 수백 자인데 한글 11,172자 전체가 담긴 통짜 폰트를 CDN에서 받고 있어, 모바일 4G 기준 첫 화면 표시가 22.8초까지 지연

해결 → unicode-range 기반 다이나믹 서브셋 버전으로 교체해 실제 등장하는 글자 조각만 받도록 전환하고, CDN preconnect 힌트 추가. 총 전송량 4,281→664KB(−85%), FCP 22.8→5.5초, Lighthouse Performance 55→66 실측 검증
